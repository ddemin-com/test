Overview

About the Project

MR Editor is a unified browser-based environment for managing, editing, and reviewing patient-specific anatomical data. The platform combines a patient workspace, medical asset library, 3D anatomical editor, and XR visualization layer into a single modular system.

My role focused exclusively on UX architecture and product design — information architecture, workflow design, interaction design, 3D editor UX, technical prototyping, and system modeling. Visual UI refinement and branding were handled separately by other designers during later production stages.

⚠️ This project is under NDA. Some details, data, and visuals have been modified or omitted to respect confidentiality agreements.

Product Vision

The product was designed as a modular medical workspace combining four core systems into a single browser-based environment.

Centralized storage for patient-related assets, sessions, and metadata. The single source of truth for every case.

📚

XR Visualization Layer

Immersive review mode for spatial inspection and collaborative analysis. Seamless transition from desktop to XR.

The long-term vision was to create a scalable platform capable of supporting collaborative surgical planning workflows.

Challenge

The Problem

Medical specialists often work across fragmented systems not designed for collaborative spatial workflows. Typical pre-operative planning involved viewing CT/MRI data in one system, editing anatomical models in another, transferring files manually, and reviewing results separately in XR environments.
🔀

Fragmented Workflows

Multiple disconnected tools required for a single preparation case. Surgeons juggle 4+ applications just to prepare for one procedure.

📂

Duplicated Data Management

Manual file transfers between systems with no version control. Assets duplicated across tools with no single source of truth.

🧊

Poor Spatial Understanding

2D scans fail to communicate complex 3D anatomy for surgical planning. Critical spatial relationships are lost in flat representations.

🧠

Increased Cognitive Load

Surgeons manage 665K+ vertices of anatomical data under time pressure. Interface complexity compounds the inherent domain complexity.

🤝

Limited Collaboration

No unified workspace for sharing 3D medical models between specialists. File sharing happens via email with no real-time review.

Research

Market Research

We analyzed existing medical and spatial systems to identify gaps and opportunities. The research covered both dedicated medical tools and general 3D editing solutions.

3D Slicer

Open source, powerful segmentation capabilities, but steep learning curve and desktop-only deployment. No collaboration features.

OsiriX

Mac-only DICOM viewer with limited 3D capabilities. No cross-platform support and no built-in collaboration workflows.

Materialise Mimics

Industry standard for medical 3D but expensive licensing. Complex workflows requiring specialist training. Desktop-dependent.

Microsoft HoloLens

MR-capable but $3,500+ per unit. Limited to Microsoft ecosystem with no integrated editing or asset management.

Blender

Powerful open-source 3D editing but not designed for medical workflows. No DICOM support, no patient data management.

CAD Editors

Technical complexity optimized for engineering, not medical professionals. Steep learning curves, no medical-specific tooling.

Key Findings

  • Fragmented tooling — most solutions required switching between independent applications
  • Technical complexity — interfaces optimized for technical specialists rather than medical workflows
  • Limited collaboration — many systems lacked centralized asset organization and shared environments
  • Desktop dependency — existing workflows heavily dependent on local workstation software
These findings influenced the decision to design a browser-based architecture with centralized patient workspaces and integrated 3D editing capabilities. We aimed to combine the best features from all analyzed tools into a single unified platform.
Complexity

Core UX Challenges

Five fundamental design challenges shaped every interaction decision in the platform.
01

Managing Complex Anatomical Hierarchies

Medical models often contained deeply nested structures and multiple mesh groups. The system needed scalable hierarchy management with grouping, visibility control, and drag-and-drop reorganization.

02

Browser-Based Spatial Editing

Large anatomical scenes introduced rendering and interaction constraints inside WebGL environments. Editing workflows had to remain responsive while handling high-polygon models with 665K+ vertices.

03

Multi-Format Medical Assets

Users needed to seamlessly work with 3D models, medical imagery, videos, and XR-ready scenes. The challenge was creating a unified navigation model across different media types.

04

Dense Professional Workflows

Medical specialists required persistent access to tools and contextual controls. The interface needed to prioritize efficiency and low interaction cost over visual minimalism.

05

XR Transition Workflows

Desktop preparation and immersive review needed to function as a continuous workflow rather than separate disconnected systems. One click from editing to spatial review.

Reflection

Key Takeaways

Spatial UX Changes Everything

Working with 3D medical data requires fundamentally different interaction thinking. Maintaining spatial orientation and preserving context became more important than visual simplification.

Hierarchy Is the Product

In complex medical scenes, the way anatomy is organized directly determines editing efficiency. Scene hierarchy architecture was the single most impactful UX decision in the entire project.

Dense Interfaces Serve Expert Users

For professional medical workflows, persistent tooling and reduced navigation depth consistently outperformed minimalist interface approaches. Experts need tools visible, not hidden.
